Detection 7 PowerShell Download Cradle

Objective
Detect PowerShell being used to download content from a remote web server.

This detection uses Sysmon network telemetry from Target-PC.
The main event used for this detection is Sysmon Event ID 3.

A PowerShell download cradle is not automatically malware.
It still needs to be reviewed because attackers often use PowerShell to pull scripts, tools, or payloads from a remote host.

Lab Setup
Host: Target-PC
Attacker: Kali Linux
SIEM: Splunk
Log Source: Sysmon
Event ID: 3
Destination IP: 192.168.10.250
Destination Port: 8000

MITRE ATT&CK
Execution: T1059.001 - PowerShell
Command and Control: T1105 - Ingress Tool Transfer

Detection Logic
Look for PowerShell network connections from Target-PC to a remote HTTP server.

In this lab, Target-PC connected to the Kali web server on port 8000.
The connection was generated by PowerShell using Invoke-WebRequest.

Common indicators:
powershell.exe
Invoke-WebRequest
iwr
http://
https://
remote IP address
unusual destination port

SPL Search
index=endpoint host="Target-PC" "192.168.10.250" "8000"
| table _time source EventCode EventID host ComputerName Image DestinationIp DestinationPort _raw
| sort -_time

Stronger Search
index=endpoint host="Target-PC" source="*Sysmon*" EventCode=3
| search DestinationIp="192.168.10.250" DestinationPort=8000
| table _time host ComputerName Image User DestinationIp DestinationPort DestinationHostname Protocol
| sort -_time

Safe Lab Simulation
A harmless text file was created and hosted on Kali using Python's built-in web server.

Kali commands:
mkdir -p /home/kali/soc-web
cd /home/kali/soc-web
echo "SOC Lab PowerShell Download Cradle Test" > test.txt
python3 -m http.server 8000

Target-PC PowerShell command:
Invoke-WebRequest -Uri "http://192.168.10.250:8000/test.txt" -OutFile "$env:TEMP\soc_lab_test.txt"

File verification:
Get-Content "$env:TEMP\soc_lab_test.txt"

Expected output:
SOC Lab PowerShell Download Cradle Test

This test was safe.
It did not download malware.
It did not execute the downloaded file.
It did not create persistence.
It did not disable security controls.
It was only used to generate Sysmon telemetry for detection validation.

Detection Result
Splunk returned Sysmon Event ID 3 from Target-PC.

The event showed:
Host: Target-PC
Source: Microsoft-Windows-Sysmon/Operational
Event ID: 3
Destination IP: 192.168.10.250
Destination Port: 8000
Process: powershell.exe

Analyst Review
The first thing I checked was whether Target-PC actually made a network connection to the Kali web server.
Sysmon Event ID 3 confirmed that the endpoint connected to 192.168.10.250 on port 8000.

The important fields were:
Host
User
Image
DestinationIp
DestinationPort
DestinationHostname
Protocol
EventID
Source

The process involved was PowerShell.
That matters because PowerShell is often used for remote file retrieval during attacker activity.

The lab file was harmless, but the behavior is still useful to detect.
In a real SOC investigation, I would not stop at the network connection.
I would check what was downloaded, where it was written, and whether anything executed afterward.

Investigation Questions I asked myself

1.Which host made the connection?

2.Which user ran PowerShell?

3.What destination IP and port were contacted?

4.Was the destination internal or external?

5.Was the destination expected?

6.What file was downloaded?

7.Where was the file saved?

8.Was the file executed after download?

9.Did PowerShell launch any child processes?

10.Were there file creation events after the network connection?

11.Was this tied to other suspicious activity?

Severity
Medium

Raise to High if:
The downloaded file is executable.
The file runs after download.
The destination is external or suspicious.
PowerShell was launched by Office, a browser, or an unknown process.
The activity is followed by persistence.
The activity is followed by credential access.
The activity is followed by Defender tampering.
The host has other suspicious PowerShell events.

False Positives
Admin scripts
Software deployment tools
Internal automation
Security testing
Lab validation
Approved PowerShell downloads
Endpoint management tools

Tuning Notes
This search is broad on purpose for lab validation.
In production, I would tune this by reviewing the destination, parent process, user context, and downloaded file.

Good tuning ideas:
Allowlist trusted internal update servers.
Allowlist approved admin scripts.
Alert higher when PowerShell contacts unusual ports.
Alert higher when the destination is external.
Alert higher when PowerShell creates a file after the connection.
Alert higher when the downloaded file is executed.
Correlate Sysmon Event ID 3 with Sysmon Event ID 1 and Event ID 11.

Recommended Response for this alert:
Identify the host and user.
Review the destination IP and port.
Confirm whether the destination is trusted.
Inspect the downloaded file.
Check the file path where the content was saved.
Search for process execution after the download.
Search for file creation events after the download.
Review PowerShell command line history if available.
Check for related persistence or defense evasion activity.
Isolate the host if malicious behavior is confirmed.
Document the full timeline.

Analyst Conclusion
This detection confirmed PowerShell download cradle behavior from Target-PC to the Kali web server.

The activity was safely simulated with a harmless text file, but the behavior is still important because attackers commonly use PowerShell to retrieve scripts, tools, or payloads from remote infrastructure. Splunk showed Sysmon Event ID 3 network telemetry, which confirmed that Target-PC connected to 192.168.10.250 on port 8000.

The main investigation point is not just that PowerShell made a network connection. The important question is what was downloaded and what happened next. In a real SOC investigation, I would review the downloaded file, check for file creation, look for child process execution, and search for any follow-on activity such as persistence, credential access, or defense evasion.
